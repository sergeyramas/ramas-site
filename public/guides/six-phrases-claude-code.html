почему порядок важнее самих фраз

Три слоя, а не список из шести пунктов

Фразы не равнозначны. Первые четыре — фундамент: скорость и предсказуемость. Пятая заставляет систему улучшать саму себя. Шестая — самая мощная и одновременно самая опасная: её включают, только когда точно понимаешь, что делаешь.

Фразы 1–4

Фундамент

Скорость и предсказуемость. Задают, как Клод дробит, планирует, уточняет и проверяет ещё до первой строки кода.

Фраза 5

Самоулучшение

Система учится на собственных граблях. Один раз исправил руками — больше эта ошибка не повторится.

Фраза 6

Мощь + риск

Полная автоматизация. Даёт максимум рычага, но кривой автомат ухудшает процесс. Нужен трезвый расчёт.

Шесть фраз в рабочем порядке

01 · фундамент

«Запускай субагентов»

По умолчанию Клод тянет всё в один диалог. Большая задача перегружает контекст — качество падает, токены горят. Субагенты дробят её на параллельные сессии: у каждой свой контекст, свой промт, свои права и даже своя модель. Ничего настраивать не надо — достаточно попросить.

> Запусти 5 субагентов на эту задачу в параллельные сессии, каждому — свою зону ответственности. copy
Когда
Только большие задачи. Мелкие — смысла нет.
3 сценария
Перспективы: 5 углов на одну проблему Масштаб: 10 агентов проверяют 10 000 доменов Скорость: параллель вместо очереди
Ловушка: из коробки Клод субагентов недоиспользует. Не скажешь явно — не запустит.
02 · фундамент

«Напиши implementation spec»

Встроенный Plan Mode слабее ручной спеки — Карпатый его не любит: сначала вместе проектируешь детальную документацию, и только потом код. Спека выбирает один путь из облака вариантов реализации. Всё, что ты не прописал заранее, Клод выдумает за тебя.

> Напиши implementation spec для [задача]. Разбей на шаги. Для каждого шага покажи ключевые решения, чтобы я мог предопределить их до сборки. copy
Эффект
Детальная спека сводит догадки Клода к 0%.
Без неё
С первого раза не попасть — переделывать по тысяче раз.
03 · фундамент

«Проинтервьюируй меня»

Какой бы детальной ни была спека, что-то ты упустишь. Дай Клоду задать вопросы: очевидные для тебя, но неочевидные для него — и наоборот, о которых ты и не думал. Ответы резко улучшают спеку.

> Проинтервьюируй меня: определи ядро (проблема, для кого, для кого нет), пройди ключевые решения, потом суммируй в implementation spec. copy
Не знаешь ответ
Так и напиши: «не знаю, предложи на своё усмотрение».
Итог
Готовый spec / PRD для запуска субагентов (фраза 1).
04 · фундамент

«Проверяй до того, как строить»

Ошибку дешевле поймать в тексте спеки, чем в готовом проекте. Чем дальше зашла сборка, тем дороже правка. Заставь Клода спланировать проверку заранее — на трёх слоях.

CLAUDE.mdПравило: «перед любой работой опиши, как проверишь» → Клод сразу строит план проверки.
ИнструментыЧем проверить: MCP для деплоя, skill-валидатор бренд-голоса, браузер — смотрит результат сам.
ЗоныHuman-in-the-loop там, где цена ошибки высока (платежи, бухгалтерия, юр. документы). Где низка (вёрстка) — быстро, не боясь сломать.
Помни: Клод даёт 99–99,5%, не ровно 100%. В критичных местах ставь человека в петлю.
05 · самоулучшение

«Сделай из этого скилл»

Повторяешь одни и те же правила из диалога в диалог — это маркер: пора паковать в скилл. У Anthropic есть Skill creator, который собирает скилл прямо из истории диалога: scripts + SKILL.md (триггер + инструкция) + examples.

> Сделай из этого диалога скилл: scripts, SKILL.md (триггер + инструкция) и примеры «как надо / как не надо». copy
2 правила
Не выдумывай абстрактно — строй из того, что есть. И только из состоявшегося диалога — кейс уже проверен руками.
Секция GOTCHAS
Грабли, на которые наступал, и правки, что делал. С каждым запуском скилл улучшает сам себя.
> На основе нашего диалога дополни любой скилл, что я использовал, секцией GOTCHAS — чтобы не повторять эту ошибку. copy
06 · мощь + риск

«Автоматизируй это»

Самая мощная фраза — и самая опасная. Кривой автомат ухудшает рабочий процесс, а не улучшает. Многие автоматизируют то, что автоматизировать не нужно. Прежде чем автоматизировать — доведи процесс почти до идеала.

> Автоматизируй это — но прогони через два фильтра, а в критичном оставь human-in-the-loop. copy
Автоматизация
AI решает сам. Годится только для доведённого, измеримого процесса.
Аугментация
AI — ускоритель, ты в контуре и перепроверяешь. Меньше авто-решений, больше контроля, выше качество.
Фильтр 1Тест на вкус: нужен вкус, чтобы оценить результат? Да → аугментация. Измеримо цифрами → автоматизация.
Фильтр 280/20: результат на 80% хорош — устроит? Да → автоматизируй. Нет → аугментируй, останься в контуре.
Красная зона: финансы, бухгалтерия, юр. документы — не доверять слепо. Только human-in-the-loop.
A/B-тест из видео · один и тот же Telegram-бот для сбора заявок в Excel

Голый Claude Code против скилла с шестью фразами

Одинаково короткая задача: «Собери Telegram-бота, который собирает заявки клиентов и заносит их в Excel». Разница — в наличии скилла, который выстраивает те самые шесть фраз в правильном порядке.

Без скилла
Один промт — и сразу код
  • Не задал ни одного вопроса — просто сделал
  • Бот примитивный: имя, телефон → строка в Excel
  • Пишет в таблицу в той же папке, новую не создаёт
  • Ничего не предложил добавить
  • Функционал ограничен ровно тем, что попросили
  • В таком виде запускать сложновато
Со Spec Kit скиллом
Интервью → спека → сборка → проверка
  • Разбил работу на этапы, не ушёл в код без подтверждения спеки
  • Задал вопросы: поля, товары кнопками, уведомления в ТГ, библиотека
  • Прислал спецификацию прямо в чат на согласование
  • Запустил 4 параллельных субагента на Opus 4.8
  • Guard-проверка: сам нашёл и исправил баги
  • Готово с первого раза: корзина, живой пересчёт суммы, «поделиться контактом», полный заказ в Excel

Итог: рабочий бот за один промт + ответы на вопросы. Без единой ручной правки кода.
